Overview

AI Companion is a mobile-first experience where users can chat, call, and bond with a personalized AI character. The companion has memory of past conversations, a consistent personality, a unique voice, and an animated avatar that reacts to emotion in real time.

Why Tencent RTC? Sub-second, cross-continent voice latency and a server-side push stream SDK that lets the AI speak back as a virtual "anchor". Combined with Tencent Cloud streaming ASR/TTS, you can keep end-to-end voice turn latency under 2 seconds.

Key features

  • Real-time voice & video calls with natural turn-taking, powered by TRTC.
  • Persistent long-term memory using Pinecone vector DB — your companion remembers birthdays, preferences, inside jokes.
  • Emotion-aware replies — every response carries an emotion label that drives avatar animation & voice tone.
  • Custom personality & voice — users can configure tone, backstory, speaking style, and pick a Tencent Cloud TTS voice.
  • Memory Journal — transparent, user-editable list of what the companion has remembered.
  • Proactive push notifications — morning greetings, "miss-you" nudges, and important-date reminders.

Architecture

The voice-call loop is the most latency-sensitive path. Here's the full pipeline:

pipeline 
  Client mic (PCM 20ms)
        ↓ WebSocket
  Backend: Tencent ASR stream  →  partial transcript
        ↓  (on sentence boundary)
  LLM (streaming, with memory-augmented system prompt)
        ↓  token stream
  Tencent TTS stream  →  PCM chunks
        ↓  trtc-server-sdk (AI "anchor")
  TRTC room  →  Client playback

Mobile client joins the room as audience. The AI push process (run as a separate worker to avoid blocking) generates the voice stream and publishes into the same room. Target end-to-end latency: < 2 seconds.

Database schema (Prisma)

Six core models:

  • User — id, nickname, avatar, timezone, lastActiveAt, pushToken, pushEnabled, createdAt
  • Companion — id, name, personality, speakingStyle, voiceId, avatarUrl, backgroundStory, ownerId
  • Conversation — id, userId, companionId, startedAt, endedAt
  • Message — id, conversationId, role, content, audioUrl, emotion, createdAt
  • Memory — id, userId, companionId, pineconeKey, summary, category, createdAt
  • LiveRoom — id, userId, companionId, roomId, status, startedAt, endedAt

Key hook — useTRTC

TypeScript 
interface TRTCHook {
  init(config: { sdkAppId: number; userId: string; userSig: string }): Promise<void>;
  joinCall(roomId: number): Promise<void>;   // joins as 'audience' — receives AI stream
  startVoiceCapture(): Promise<void>;          // opens mic, streams PCM to backend via WS
  endCall(): Promise<void>;
  state: 'idle' | 'connecting' | 'in-call' | 'error';
  networkQuality: number;                     // 1 (best) – 6 (worst)
}

Call flow:

  1. POST /api/trtc/token → receive { sdkAppId, userId, userSig, roomId }
  2. Join room as audience
  3. Open local mic; send raw PCM (20 ms frames) over WebSocket to backend
  4. Backend processes audio → AI voice stream returned via TRTC anchor bot
  5. Subscribe to remote stream; auto-play
  6. If networkQuality ≥ 5, auto-degrade to text-only mode

Backend APIs

Chat pipeline — POST /api/chat/message:

  1. If voice input: Tencent ASR → text
  2. Embed user message (OpenAI text-embedding-3-small)
  3. Pinecone top-5 memories filtered by userId + companionId
  4. Build system prompt (template below) + call LLM (streaming, JSON mode)
  5. Tencent TTS on content → upload MP3 to COS → audioUrl
  6. Save Message → return to client
  7. Async: memory extraction job

TRTC routes:

  • POST /api/trtc/token — generate userSig using tls-sig-api-v2 (server-side only).
  • POST /api/trtc/end — mark room ended, save transcript, trigger memory update.
  • WebSocket /ws/call/:roomId — pipe mic PCM → ASR → LLM → TTS → TRTC push stream.

Critical implementation notes

  1. UserSig is server-side only. Never ship TRTC_SECRET_KEY to the client.
  2. AI push stream runs as a separate worker. Do not block the main API process.
  3. Streaming everywhere. Use streaming ASR + streaming LLM to stay under 2-second turn latency.
  4. Graceful degradation. networkQuality ≥ 5 → auto fallback to text-only.
  5. Privacy by namespace. Pinecone vectors scoped by userId.
  6. Push rate limiting. Max 3 pushes/user/day.
